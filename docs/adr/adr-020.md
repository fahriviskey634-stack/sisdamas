# ADR-020: Future Scalability (Project Scoping)

*   **Status:** Approved
*   **Date:** 2026-07-14
*   **Authors:** Chief Software Architect, Enterprise Architect

## Context
The platform is designed for a single KKN group, but should be reusable.

## Problem Statement
How to ensure future KKN teams can reuse the code?

## Decision Drivers
Reuse limits, database schema.

## Alternatives Considered
Project Table Scoping (Selected) vs. Single-village lock-in.

## Decision
Include a `project` table structure from day one to scope all data entities.

## Advantages
Code is reusable for other villages without schema rewrites.

## Disadvantages
Minor data join overhead.

## Trade-offs
Accept minor join overhead in exchange for long-term scalability.

## Consequences
All tables contain a `project_id` reference.

## Rejected Alternatives
Single-village lock-in (rejected as it would require a database schema rewrite in future KKN periods).

## Risks
Query complexity.

## Mitigation
Setup database indexes on project reference columns.

## Future Considerations
Setup multi-tenant dashboard templates.

## References
`06_DATABASE_SPECIFICATION.md`.

## Review Notes
Master tables include project reference columns.

## Lessons Learned
Design database schemas with scalability in mind.
